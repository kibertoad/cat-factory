## Goal & rationale

Two problems in `frontend/app` (`@cat-factory/app`), one library that addresses both, and a deliberate dual goal.

**Problem 1: the extensibility ceiling.** Everything pluggable in the SPA today is data-driven from the backend snapshot: custom agent kinds (`useAgentsStore().registerCustomKinds`), provider backend kinds, initiative presets, skills. A consumer deployment (`deploy/frontend` is a ~5-file `extends` shell) can rebrand, retheme, and override locale strings, but it cannot contribute _components_: a new step result view (`STEP_RESULT_VIEWS` in `panels/StepResultViewHost.vue` is a hardcoded record), a navbar or command-palette entry (`SideBar.vue` / `CommandBar.vue` are hand-written item lists), an inspector panel (`InspectorPanel.vue` switches on `block.level` with `v-if` sections), or a modal (`pages/index.vue` mounts ~50 of them by hand). A company wanting a proprietary frontend extension for its custom agents today has exactly one option: fork the layer. The backend solved the equivalent problem with public registries (`registerAgentKind`, `registerGate`, the [registry DI migration](./registry-di-migration.md)); the frontend has no counterpart.

**Problem 2: hand-rolled structure in the standardization targets.** The areas the platform keeps growing are each reinvented per feature:

- **Navigation.** `SideBar.vue` (439 lines) hardcodes every nav button behind per-item RBAC computeds; `CommandBar.vue` re-derives the same catalog and the same gating by hand; `BoardToolbar.vue` is a third hardcoded list. Adding one destination means editing all of them plus the `ui` store and `index.vue`.
- **Wizards.** There is no shared multi-step framework. `EnvironmentSetupWizard.vue` (675 lines) plus its dedicated 489-line store roll their own `STEP_ORDER`; `BootstrapModal.vue` (758 lines), the onboarding modals, and the credential flows each invent their own step tracking and header markup.
- **Detail panels.** `InspectorPanel.vue` (631 lines) is a level-switched monolith importing ~20 sub-panels; only `InspectorSection.vue` is shared structure.
- **Agent-run details.** The big result windows (`RequirementsReviewWindow` 1220, `TestReportWindow` 931, `AgentStepDetail` 913, `ObservabilityPanel` 740, `PipelineProgress` 695) share good primitives (`StepRunMeta`, `StepRestartControl`, the `useResultView` contract) but each duplicates its dialog chrome (Teleport, backdrop, header, close handling).

**The dual goal.** [modular-vue](https://github.com/kibertoad/modular-react) (`@modular-vue/*`, developed in the modular-react monorepo, local checkout at `C:\sources\modular-react`) is our own library, and this adoption is explicitly a co-evolution program: cat-factory gets a real module/registry/slot/journey architecture, and modular-vue gets the production-consumer pressure that matures it, the same role `docs/consumer-feedback-production-app.md` played for the React family. Improving the library is part of every slice, not a precondition to clear once.

## What modular-vue offers (the primitives this initiative uses)

- **Module descriptors + registry.** `defineModule({ id, version, navigation, slots, dynamicSlots, requires, entryPoints, ... })` describes a feature as a plain object; `createRegistry(config)` composes registered modules into an application manifest (navigation manifest, slots manifest, DI wiring), validating duplicate ids, missing dependencies, and semver `moduleCompat` ranges at resolve time.
- **Slots.** Named arrays concatenated across all modules; `dynamicSlots(deps)` factories plus a global `slotFilter` re-evaluate on state change (`recalculateSlots()`), which is exactly the shape of RBAC-gated nav/command items.
- **Journeys.** Typed, serializable multi-module workflows: entry/exit point contracts, transitions, back/rewind, pluggable persistence, URL sync (`useJourneySync`), and Vue mount adapters that host a journey outside route navigation (`@modular-vue/journeys`).
- **Remote capability manifests.** `RemoteModuleManifest` / `mergeRemoteManifests`: backend-delivered JSON drives slots and navigation, data only, no components over the wire. This is a formalized version of the `customAgentKinds` snapshot pattern cat-factory already hand-rolled.
- **DI between modules.** Registry-owned `stores` / `services` / `reactiveServices` buckets; modules declare `requires` / `optionalRequires` and consume via `createSharedComposables` (`useStore`, `useService`, `useSlots`, `useNavigation`).
- **Nuxt integration.** `@modular-vue/nuxt` installs the resolved manifest on the Nuxt app from a plugin. cat-factory is `ssr: false`, which is the binding's simple case (singleton registry, no per-request state).

Out of scope for this SPA: the route-centric surface (`createRoutes()`, route-driven zones) and the compositions engine (0.1.x). cat-factory is a single-route board app; the value here comes from registry, slots, journeys, remote manifests, and DI, all of which work for non-routed modules.

### Weak or neutral

- **Route-centric surface goes unused.** Modules here will mostly be non-routed (no `createRoutes`), and route-driven zones do not apply to a single-route board. Unused surface, not a blocker; if deep links ever become routes (see `global-search-and-deep-links.md`), the surface is waiting.
- **Compositions** (`@modular-frontend/compositions-engine` 0.1.x) are not needed for any target area. Excluded from scope.

### Known upstream gaps (the opening entries of the upstream backlog)

These are not static risks; per the co-evolution model below they are the first upstream work items, owned by this initiative.

1. **Pinia interop.** The `Store<T>` contract is zustand-shaped (`getState`/`setState`/`subscribe`); cat-factory has 70 Pinia stores. A thin adapter (Pinia `$state`/`$subscribe`/`$patch` behind the contract) is feasible and is already a tracked follow-up in the library (`docs/vue-support-tracker.md`, decision D3). Ship it upstream, not as a cat-factory shim.
2. **`@modular-vue/nuxt` maturity.** The binding is 0.1.0 experimental. cat-factory's `ssr: false` layer is its easy case, but "registry factory exposed through a Nuxt layer's `extends` chain so the consumer can contribute modules" is a consumer story the library has not exercised. Harden and document it upstream; graduate the package out of 0.x as part of slice 0/1.

### Adoption risks on the cat-factory side

- **Migration scale.** ~180 components (~49k lines), 70 stores. Big-bang restructure is a no-go; this plan is strangler-only, seam by seam, matching how every other migration in this repo runs.
- **i18n guards.** Typed message keys (`typedOptionsAndMessages`) plus the parity scripts gate CI against one shared catalog per locale. Internal modules keep their strings in the shared catalog; consumer extension modules ship their own locale JSON through the existing layer deep-merge, which already supports exactly this.
- **Thin test coverage.** 33 spec files, no component tests. Every slice needs e2e coverage (`data-testid` + live-push assertions per the e2e rules) before its refactor, not after.
- **Choke points stay out of scope.** `workspace.applySnapshot` (fans into ~20 stores) and `useWorkspaceStream.onMessage` (17-branch switch) are real coupling, but modularizing the event fan-out is a separate future initiative. Touching them here would balloon every slice.
- **Maintenance.** modular-vue is single-maintainer, but the maintainer is us; the usual third-party abandonment risk is instead a dogfooding feedback loop, which is precisely the point.

## Recommendation: conditional GO

Adopt modular-vue in `@cat-factory/app` as a phased strangler migration of the non-route primitives (registry, slots, journeys, remote manifests, DI). The conditions are about _how_, not _whether_:

1. **The co-evolution loop is honored every slice.** A gap found mid-slice is fixed upstream in modular-vue and the released improvement re-adopted before the slice closes. Local shims in cat-factory never outlive their slice; a slice with a lingering workaround is still open.
2. **Strangler-only.** Each slice converts one seam and leaves the rest untouched. No big-bang, no parallel rewrite.
3. **Route-module restructure and compositions are out of scope.** Revisit only if the app grows real routes.

## Target pattern

Two halves, both mandatory per slice.

### The cat-factory shape

- **Registry factory in the layer.** `@cat-factory/app` owns a registry factory that registers the first-party modules and resolves the manifest from a Nuxt plugin (`@modular-vue/nuxt`, or its runtime `installModularApp` directly until the module wrapper matures). `ssr: false` means a singleton registry is fine.
- **Consumer contribution seam.** A deployment extending the layer contributes its own modules through a well-known seam in the extends chain (the exact mechanism, config-referenced module list vs. consumer plugin, is a slice 0 decision and an upstream documentation deliverable). Extension modules are first-class: nav items, command entries, result views, panels.
- **First-party features become modules incrementally.** Each converted area (navigation, result views, wizards, inspector panels) is registered through the same seam the consumers use, dogfooding the extension story exactly the way `@cat-factory/gates` dogfoods `registerGate` on the backend.
- **Backend data keeps driving capability.** The snapshot's `customAgentKinds` (and future capability lists) arrive as remote capability manifests merged into slots, replacing the ad-hoc `AGENT_BY_KIND` mutation while keeping the "backend-only change lights up a new capability" property.

### The co-evolution loop (the per-slice working protocol)

For every slice:

1. **Adopt** the primitive in the slice's target area.
2. **Reflect**: write down what fit cleanly and what needed bending (in the slice's PR description and this tracker's checklist row).
3. **File upstream**: every bend becomes an issue or change in the modular-react repo: a new API, an ergonomics fix, a doc gap, missing Vue/Nuxt parity. Feedback lives in the upstream repo, not only in this tracker.
4. **Land and release** the upstream change (`@modular-vue/*` release).
5. **Re-adopt immediately**: bump to the released version in the same slice and delete any interim shim.

A slice is `done` only when cat-factory runs on the released upstream improvements. The checklist records upstream outcomes (issues, releases consumed) next to the cat-factory PRs so the co-evolution half of the work is as visible as the migration half.

**The standard upstream mechanism (applies to every slice, not just this one).** Any upstream change is authored as a real PR in the modular-react repo — code + tests + docs, on its own branch — carrying a release label (`patch` / `minor` / `major`; modular-react auto-bumps + publishes on merge via that label, it does not use changeset files). While that PR is unreleased, cat-factory develops the slice against the new API via a temporary local link (a pnpm `file:` override in the root `package.json`), so the cat-factory side is fully buildable and tested before the release. Once the upstream release lands, bump the `@modular-vue/*` pin to the published version and delete the `file:` override in the SAME slice. The changeset lives on the cat-factory side (`@cat-factory/app`); the upstream side uses the label. This is the durable workflow for all remaining slices.

## Slice 0 outcomes

- **Consumer contribution seam (decided).** The layer owns `app/modular/registry.ts` — `createAppRegistry()` builds a `@modular-vue/runtime` registry, registers the first-party modules, then everything a consumer contributed. A deployment extending the layer calls the auto-imported `registerAppModule(...)` from its own plugin. Module descriptors carry Vue components, so they can't travel through serializable Nuxt config; this in-process array seam is how the layer stays unforked while a consumer contributes real components. We use `installModularApp` from a hand-written plugin (Option B), NOT the library's serializable Nuxt module (Option A): Option A's `registry: '~/modular/registry'` convention resolves `~/` to the _consumer's_ srcDir under `extends` (so the layer's base modules and consumer additions don't compose), and it can't take the `slotFilter` the RBAC gating in slice 1 needs.
- **Plugin ordering (the load-bearing detail).** The layer's install plugin is `enforce: 'post'`. Nuxt loads layer plugins before the consuming app's plugins within one enforce bucket, so a consumer registering from a normal plugin would otherwise run after the layer resolves and be missed. `post` flips it: the consumer's default-bucket registration runs first, then the layer resolves. A consumer contributes from a default (or `pre`) plugin.
- **`vue` pinned up, not the peer relaxed.** cat-factory raised its workspace `vue` pin `3.5.39 → 3.5.40` (the `@modular-vue/*` peer floor) rather than relaxing the upstream peer — one singleton, satisfied cleanly. `@modular-vue/*` / `@modular-frontend/*` added to `minimumReleaseAgeExclude` (namespaces we own).
- **No upstream code change was needed to install** — with the pin raised, cat-factory consumes the published `@modular-vue/*` directly (no local link). The co-evolution half is docs + one peer widen (below), not a blocking release.
- **Co-evolution tail: CLOSED.** cat-factory is on `vue-router@5.x` (Nuxt owns it) while the first-published `@modular-vue/{core,runtime,nuxt}` peer-required `vue-router@^4.5.0` — a non-fatal peer warning (our zero-route Option-B path never touches the router at runtime). Fixed upstream in [modular-react#87](https://github.com/kibertoad/modular-react/pull/87) (peer widened to `^4.5.0 || ^5.0.0`) and re-adopted here: `@modular-vue/core@^1.0.1`, `@modular-vue/runtime@^1.0.1`, `@modular-vue/nuxt@^0.1.1`. `pnpm peers check` is clean. No shim ever lived in cat-factory, so the slice is fully closed on both halves.

## Slice 1 outcomes

- **One catalog, three shells.** `app/modular/nav-contributions.ts` declares every nav/command destination once as data (id, icon, i18n keys, `surfaces`, a reactive `gate` predicate, an `action` id or consumer `run` closure, per-surface placement). `SideBar`, `CommandBar`, and `BoardToolbar` render from it via `useNavContributions()`, replacing three hand-maintained item lists + per-item RBAC computeds. The dynamic per-connection integration commands (GitHub/Slack/document/task connect) stay local to the palette — they vary per live connection, not part of the static manifest.
- **Reactive RBAC gating via the new upstream primitive.** Gating is a `slotFilter` over a reactive `gates` service (`app/modular/nav-gates.ts`, getters over `useWorkspaceAccess()` + stores), read through `useReactiveSlots` (modular-react#88). An item shows/hides the instant a permission or connection flips, with no `recalculateSlots()` — the ergonomics the signal path couldn't give cleanly for reactive RBAC.
- **What fit cleanly / what bent (the reflection half).** Fit: slots + a reactive `slotFilter` model host-owned RBAC exactly; the consumer `nav`-slot seam needed zero new surface. Bent (fixed upstream, not shimmed): the signal-only slot read forced either a `watchEffect(recalculateSlots)` workaround or an upstream reactive binding — chose the latter (`useReactiveSlots`, modular-react#88), so no shim lives here. Two type-friction notes for later slices: a registry `TSharedDependencies` must be a `type` (not `interface`) to satisfy `Record<string, any>`, and an `AppSlots` index signature must be mutable (`unknown[]`) to satisfy the runtime `SlotMap` while still meeting `useReactiveSlots`' `readonly` bound.
- **Two intentional behaviour unifications** (noted in the changeset): account settings gates on `accountsEnabled` in both shells (the palette previously showed it unconditionally); one icon per destination across shells. Everything else — including the command-palette group and within-group ordering — is exact 1:1 parity with the pre-slice-1 shells (pinned by `nav-contributions.spec.ts`).
- **First-party actions are a typed, exhaustive contract.** A catalog item's `action` is a `NavActionId` union (`NAV_ACTIONS`), and `useNavContributions` resolves it against a `Record<NavActionId, () => void>` handler map — so drift between the catalog and the handler map is a compile error, not a silently dead button. Consumer modules stay on their own `run` closure. The pure grouping/ordering helpers (`groupSidebar`/`groupCommands`/`sortToolbar`) live in `nav-contributions.ts` (no Vue/Nuxt deps) so they're unit-tested directly; the composable is a thin reactive wrapper.
- **Re-adopted on release (co-evolution loop closed).** #88 merged with a review refactor — the runtime now resolves ONE shared `computed` at install (`resolveReactiveSlots` + `reactiveSlotsKey`; `useReactiveSlots` is a thin reader injecting it, so evaluation happens at most once per change across all consumers) instead of the per-component computed the original PR provided via `reactiveSlotsConfigKey`. The public `useReactiveSlots<T>()` signature is unchanged, so cat-factory needed no code change. Released as `@modular-vue/vue@1.1.0` + `@modular-vue/runtime@1.1.0`; the pins in `frontend/app/package.json` are bumped and the temporary `file:` override is deleted. No shim ever lived in cat-factory — the slice is fully closed on both halves.

## Slice 2 outcomes

- **Result views are a modular registry, not a hardcoded `Record`.** Every built-in window is a `ComponentEntry` in the first-party `resultViews` slot (`app/modular/result-views.ts`); `StepResultViewHost.vue` reads the merged slot via `useReactiveSlots` and indexes it with the upstream `resolveComponentRegistry`. The old `STEP_RESULT_VIEWS` map is gone. A consumer deployment contributes its OWN window to the SAME slot via `registerAppModule` (paired against the kind's `presentation.resultView` id) with zero host edits — the slice-2 extensibility promise, dogfooding the exact seam consumers use.
- **Custom agent kinds flow through the modular system; `AGENT_BY_KIND` is frozen.** Backend-registered kinds are modeled as a per-workspace `RemoteModuleManifest` (`agents` store `hydrateCustomKinds`, read directly per the guide's single-active-manifest advice — `mergeRemoteManifests` is for holding many at once, which we don't); CODE-shipped consumer kinds enter via a static `agentKinds` slot (`registerConsumerKinds`, fed once from the resolved manifest by the install plugin). The store merges built-ins + consumer + backend + runtime into a reactive catalog and projects it into a `catalog.ts` read-model (`setCustomAgentKindMeta`, a sync-flush `shallowRef`) so the ~17 pure `agentKindMeta` / `isKnownAgentKind` callers resolve custom kinds reactively **without** the removed `registerCustomKinds` mutation of the built-in const.
- **The pairing primitive was the upstream half — released and re-adopted (co-evolution loop closed).** The spec ([`modular-vue-slice2-upstream-pairing.md`](./modular-vue-slice2-upstream-pairing.md)) shipped essentially as written: `@modular-frontend/core@0.2.0` adds `resolveComponentRegistry` / `pairById` / `ComponentEntry` (+ a `componentPairingPlugin` for static refs), re-exported from `@modular-vue/core@1.1.0`, plus a `remote-capability-manifests.md` pairing guide. cat-factory imports them only through the `@modular-vue/*` binding (never the neutral engine), bumped the pins, and kept **no** shim. `pairById`'s `missing` bucket replaces the host's hand-rolled dev-warn for a dangling `resultView` id.
- **Backend picklist opened (cat-factory-local, not upstream).** `agentPresentationSchema.resultView` (`@cat-factory/contracts`) now accepts a built-in id OR a consumer-namespaced id (`<ns>:<name>`); the boot-time registration validator (`validateRegistrations`) matches. A bare non-built-in id still fails (the typo guardrail). This is what lets a BACKEND-registered custom kind select a CONSUMER frontend view — the guide's headline case — end to end.
- **What fit cleanly / what bent (the reflection half).** Fit: the two-slot pairing (data `agentKinds` × component `resultViews`, joined by id) is exactly the released primitive's shape; consumer extensibility needed zero new host surface. Bent (fixed upstream, not shimmed): the remote-manifest helpers lived only in the neutral engine and the guide stopped short of the pairing — both closed by the 0.2.0 release. Bent (worked around, noted): a Pinia store can't call `useReactiveSlots` (no inject context), so custom kinds reach the store via the install plugin feeding the resolved static slot + the snapshot manifest, rather than the store reading the reactive slot directly — a host wiring choice, not a library gap. ## Slice 3 outcomes (landed)

All `env-setup-*` `data-testid`s are preserved.
- **Pinia-backed resume.** The journey registers `createPiniaJourneyPersistence` keyed by the target frame (`environment-setup:<frameId>`), so closing and reopening the wizard for the same frame RESUMES at the step it was left on (and completing clears the blob so the next open is fresh) — a UX gain the reset-on-every-open store couldn't offer. Session-scoped (not wired through `pinia-plugin-persistedstate`), so a full reload starts fresh.
- **Journey wiring rides the registry-plugin seam.** `createAppRegistry` attaches `journeysPlugin()` (`.use(...)`), the client plugin registers the journey + its step module and calls `provideJourneyRuntime(vueApp, manifest.journeys)` so `<JourneyOutlet>` resolves the runtime from context. The pure journey graph lives in `environmentSetup.logic.ts` (unit-tested); the `.vue`-importing wiring in `environmentSetup.ts` is registered from the client plugin (like `result-views.ts`), keeping the unit-tested `registry.ts` graph SFC-free.
- **Dependency hygiene (the bump's collateral).** Nuxt `4.4.8 → 4.5.0` pulls `vue-router@^5.2.0`; a `vue-router: 5.2.0` pnpm override keeps it a single singleton (`@nuxtjs/i18n` still resolved `5.1.0`). Pinia bumped `3 → 4` (`@pinia/nuxt@1.0.1`) alongside. Single versions verified for `vue` / `vue-router` / `pinia`. `nuxt` + `@nuxt/*` are a documented, owner-authorized `minimumReleaseAge` exception (same-day 4.5.0 release) to prune once aged out.
- **What bent (the reflection half) — two small binding-surface findings to file upstream, not blockers:** (1) the entry/exit authoring helpers `defineEntry` / `defineExit` are exported only from the neutral `@modular-frontend/core`, not re-exported by `@modular-vue/core` (so authoring a step module reaches past the binding — mirrors slice 2's remote-manifest Gap A); we consume `defineExit` from the engine and author entries as plain literals (which also sidesteps `defineEntry` widening `buildInput` to optional). (2) `installModularApp` types the registry's plugin tuple as `any`, so the resolved manifest's extension type is erased — `manifest.journeys` needs a cast to recover the `JourneyRuntime`. Both are additive typing improvements to request; the runtime values are correct today.
- **Scope closed on the real wizard surface (the re-scope decision).** The slice targeted "then `BootstrapModal` and onboarding flows," but a whole-codebase survey (every `STEP_ORDER`/`currentStep`/`stepIndex`/`goToStep`/`nextStep`/stepper token across `components/` + `stores/`) found the environment-setup wizard is the **only** component with genuine forward/back step navigation. The named follow-on targets are single-screen and were deliberately NOT force-converted:
  - `BootstrapModal.vue` (758 lines) — one launch form with a `mode` radio toggle (`reference`/`scratch`) revealing a config section, plus an inline reference-architecture CRUD sub-form. No step index, no back/next.
  - `AiProviderOnboardingModal.vue` (109 lines) — a static list of three route cards; each CTA closes this modal and opens a different panel. No steps.
  - `GitHubOnboarding.vue` (44 lines) — a connect gate wrapping `<GitHubConnect>`. No steps.
  Wrapping a single-screen form in a journey (one entry, no transitions) is pure ceremony with no navigation/resume benefit — it would add a module + entries + exits + a host for zero user-visible gain, against the repo's "clean architecture over quick solutions" principle. Journeys remain available if any of these later grows a real multi-step flow. **Slice 3 is closed on the one wizard that warranted it.**

## Slice 4 (inspector panels) — BLOCKED on upstream (`@modular-vue/zones`)

- **The next stage is the `InspectorPanel.vue` monolith** (631 lines): a fan of `v-if` blocks keyed off the selected block's `level` (frame/module/task/epic/initiative) + secondarily `type` (frontend/service) + live run state, mounting ~19 sub-panels. It is the app's biggest hand-rolled detail surface and the natural registry target.
- **Its clean solution needs a primitive the Vue binding lacks — a subject-keyed zone.** modular-vue zones are **route-driven** (contributions selected by the active route); the inspector's contributions are selected by a runtime **subject** (the selected block) — a piece of app state, not a URL. Plain slots (slice 1) gate globally and thread no subject to the predicate or the rendered component, and slice 2's `pairById` is pick-**one** where the inspector renders **all** matching panels. Hand-rolling a subject-keyed multi-panel host in cat-factory would be a local re-implementation of a missing zone concept — a shim that outlives its slice, which this initiative forbids.
- **Per the co-evolution protocol, the upstream half is filed first.** Spec: [`modular-vue-slice4-upstream-zones.md`](./modular-vue-slice4-upstream-zones.md) — a new `@modular-vue/zones` binding (`defineZone<Subject>` / `ZoneOutlet` / `useZone` / `useZoneSubject`, route-free, subject-reactive, module-contributed + consumer-extensible, a `/testing` entry, and the `installModularApp` provide + peer-range widen). It also validates against the real `InspectorPanel` shape (§5) and notes slice 5 (agent-run windows) reuses the same primitive keyed by the selected step. **Blocked until the release; the cat-factory side lands on re-adoption, exactly as slices 2 and 3 did.**

## Conventions & gotchas

- **No local shims outliving their slice.** The fix for a library gap goes upstream and is re-adopted before the slice closes; the tracker row records the release that closed it.
- **Feedback lives upstream.** File issues/PRs in the modular-react repo; this tracker only summarizes outcomes.
- **Version coordination.** The neutral `@modular-frontend/*` engine packages pin tight peer ranges, so an engine bump means coordinated releases of the dependent bindings; follow the versioning policy in the library's `docs/vue-support-tracker.md`.
- **i18n:** internal modules keep keys in the shared catalog (typed keys + parity gates stay intact); consumer extension modules ship locale JSON via the layer deep-merge. Never move first-party strings out of `i18n/locales/`.
- **e2e first.** Each slice adds `data-testid` coverage and live-push e2e specs for its target area before refactoring it, per the e2e conventions in `CLAUDE.md`.
- **No route modules.** Modules here are non-routed; do not introduce `createRoutes()` usage without a real routing initiative.
- **Leave the choke points alone.** `workspace.applySnapshot` and `useWorkspaceStream.onMessage` are out of scope until a dedicated event-fan-out initiative.
- **Changesets:** every slice touching `@cat-factory/app` needs a changeset; consumer-visible seam changes are minor bumps.
