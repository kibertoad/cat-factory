# Workspace RBAC — security follow-ups

**Status:** in progress (SEC-RBAC-0 + SEC-RBAC-5 landed) · **Owner:** core · **Started:** 2026-07-19

> Durable source of truth for the residual security work surfaced by an adversarial review of
> the shipped [`workspace-rbac`](./workspace-rbac.md) initiative (slices 1–9). Read that tracker
> first for the design; this one only carries what the review found still-open. It is a companion
> to [`security-hardening-round-2.md`](./security-hardening-round-2.md) (the RBAC review reuses its
> "re-authorize secondary ids at every boundary" target pattern).

## Goal & rationale

A focused security review of the workspace-RBAC enforcement path — the gate
(`server/src/http/authGate.ts`), the resolution logic
(`kernel/src/domain/workspace-access.ts`), the `workspaceAccess` cache slice, the member/account
services, and every controller mounted under `/workspaces/:workspaceId` **and** the `/`-mounted
side doors — was run to find authorization bypasses, cross-workspace IDOR, and cache-coherence
holes.

**The enforcement path is strong; the one exploitable finding is an account-tier confused
deputy, NOT a workspace-gate break.** The review found a single **High** cross-tenant
vulnerability (`SEC-RBAC-0`) in an **account-scoped** route where a body/query-supplied
`viaWorkspaceId` is used to fetch through another tenant's stored document-source credentials
with no ownership check — which lands _outside_ the `/workspaces/:workspaceId` gate's coverage and
is a concrete instance of the "account tier has no central gate" root cause (`SEC-RBAC-2`). The
workspace-scoped surface itself is clean:

- **No cross-workspace IDOR** in any `/workspaces/:workspaceId` controller. Every controller threads the
  **route** `workspaceId` into its service/repo lookups; every repository method that takes a
  `workspaceId` binds `workspace_id` in its `WHERE` (verified across ~460 workspace-scoped methods
  on **both** D1 and Drizzle, plus the Node non-Drizzle repos). A foreign resource id paired with
  WS-A's path resolves to `WHERE workspace_id = 'A' AND id = '<B-resource>'` → no row → 404
  (fail-safe). The three genuinely unscoped lookups (`BlockRepository.findById`,
  `ServiceRepository.get`, `PublicApiKeyRepository.getById`) are each explicitly guarded (mount
  check / account-equality check / key-auth path only). Body-supplied relationship ids
  (`reparent.parentId`, `toggleDependency`, `assignToEpic`) resolve **both** endpoints via the
  workspace-scoped `resolveBlock`.
- **No gate bypass.** `PUBLIC_PREFIXES` is separator-anchored (`/api-keys` ≠ `/api/`), the WS
  upgrade bypass is `$`-anchored on `/events` and the handler still ticket-authorizes, the viewer
  write floor covers every non-GET/HEAD method (only the read-equivalent ticket mint is exempt),
  and no GET route mutates.
- **Sound resolution + list filtering.** `resolveWorkspaceAccess` fails closed on every branch
  (legacy-owner, non-account-member with a stale row, restricted-no-row), and `listVisible`'s
  four-branch predicate agrees with it (a restricted board an account member has no row on is both
  denied by the gate and absent from their list).
- **Correct cache invalidation for every _existing_ write path** — roster/access-mode writes drop
  the workspace group; account-tier membership writes (`addMember`/`setMemberRoles`/invite-accept)
  drop everything (`invalidateAll`).

So this tracker is **one High confused-deputy fix + residual hardening + regression-guarding + one
latent trap**. `SEC-RBAC-0` is the one to land first (it exfiltrates cross-tenant document content
and misuses another tenant's stored integration secret); the rest is match-to-severity. When the
committed scope
is closed, fold the load-bearing items into the workspace-rbac closing ADR (slice 10) rather than
leaving a second tracker.

## Target pattern

- **A workspace/account id taken from the BODY or QUERY must be re-authorized against the gated
  scope before its secrets or data are touched** — the workspace gate only authorizes the id in the
  URL PATH. `SEC-RBAC-0` is the reference miss: an account-tier route accepts a `viaWorkspaceId` in
  the body and fetches through that workspace's stored credentials with no check that it belongs to
  the caller's account or that the caller can access it. This is the same class as
  `security-hardening-round-2` SEC-1 (`accountOf(id) === :accountId`) — grep for every
  body/query-supplied `*WorkspaceId` / `*AccountId` and prove each is re-checked.
- **Invalidation is the coherence story, not the TTL.** Any new write that changes a caller's
  effective access must drop the `workspaceAccess` cache after the commit — `invalidateGroup(ws)`
  for a board-scoped change, `invalidateAll()` for an account-tier change that fans across boards.
  A raw-repository write that bypasses `WorkspaceMemberService` / `AccountService` skips the
  invalidation — route removals through a service, never a bare repo `remove`. (`SEC-RBAC-1`.)
- **Every trust tier gets ONE gate, not per-handler diligence.** The workspace tier already has it
  (`mountAuthGate` + floor). The account tier does not — that asymmetry is the most likely place a
  future regression lands (`SEC-RBAC-2`).
- **The wire projection is a security boundary.** Where a controller lets reads through to all
  roles (`requireWorkspacePermission` gates writes only), the `toWire`/summary projection is what
  keeps secrets from a viewer — it must stay secret-free by construction, guarded by a test, not by
  reviewer memory (`SEC-RBAC-3`).
- **Scoping conventions need a conformance guard.** The IDOR-safety is a ~460-method convention with
  no automated assertion that a foreign-workspace id 404s; add one so a future unscoped query fails
  a test instead of shipping (`SEC-RBAC-6`).

| SEC-RBAC-0 | **High** | **Cross-tenant confused deputy via body-supplied `viaWorkspaceId` (account-tier document fragments).** The account-scoped fragment routes `POST /accounts/:accountId/document-fragments` (`createDocumentFragment`) and the account-scoped `refresh` accept a `viaWorkspaceId` from the request **body / query** and pass it straight to `documentResolver.fetch(viaWorkspaceId, …)`, which loads **that workspace's** decrypted document-source credentials (Confluence/Notion/GitHub) via `DocumentConnectionService.requireConnection` — a pure data load with **no authorization** — and fetches the caller-chosen document, storing its full body on a fragment in the caller's OWN account. Dependencies: **SEC-RBAC-0 lands first** (the only actively exploitable finding). It and
SEC-RBAC-2 share the same root cause (no account-tier gate / body-supplied ids un-reauthorized), so
fixing SEC-RBAC-0 with the shared `loadWorkspaceAccess` helper is the pilot for SEC-RBAC-2's fuller
account-tier gate. SEC-RBAC-1 and SEC-RBAC-2 are the other load-bearing ones (both add a conformance
assertion; SEC-RBAC-2's fuller form is the deferred `AccountPermission` work from workspace-rbac
§10). SEC-RBAC-3/4/5/6 are independent small hardening PRs. SEC-RBAC-7 is the events-hub follow-up
already named in the parent tracker. Every fix lands in a **shared package** (`server`, `kernel`,
`workspaces`, `caching`, `conformance`) or a single facade's config, so cross-runtime symmetry is
preserved for free — the exception is SEC-RBAC-1's removal endpoint, which touches the account-tier
service + a conformance assertion on both facades.

### Landed (this PR)

- **SEC-RBAC-0** — `FragmentLibraryController` now re-authorizes the account-tier `viaWorkspaceId`
  (body on `createDocumentFragment`, query on `refresh`) through `requireViaWorkspaceAccess`: it
  requires `workspaceService.accountOf(viaWorkspaceId) === :accountId` **and** the caller passes
  `loadWorkspaceAccess` on that workspace, else 404 (existence hidden) — before any
  `documentResolver.fetch`. The fix lives entirely in the shared `@cat-factory/server` layer both
  facades mount, so it is runtime-symmetric by construction; the assertion is a server-package unit
  test (`test/fragmentLibraryViaWorkspaceAuthz.spec.ts`) driving the real controller with a faked
  container (the auth-enabled account-scoped path the dev-open facade integration suites can't
  reach), rather than a per-facade conformance test.
- **SEC-RBAC-5** — the auth gate wraps `decodeURIComponent` of the `:workspaceId` segment in
  try/catch and returns the 404 not-found shape on a `URIError`, so a malformed
  `/workspaces/%zz/...` fails closed as 404 instead of an opaque 500
  (`test/authGate.spec.ts`).
